# CloudTool2 项目结构重构设计方案

> 版本：v2.0
> 日期：2026-04-01
> 状态：草案

---

## 1. 背景与目标

### 1.1 现状

CloudTool2 是一个基于 Qt5、VTK 和 PCL 构建的三维点云处理应用程序。当前架构存在以下问题：

- **算法层无法独立使用**：想调用一个滤波算法，必须安装 Qt + VTK
- **核心数据结构绑死 Qt**：点云对象用 `QString` 做 ID、用 `QMap` 做标量存储
- **算法类继承 QObject**：7 个算法类全部用 Qt 信号槽通信，脱离 Qt 无法编译
- **编译耦合严重**：UI 基类传递依赖整个 VTK 栈，修改一个头文件触发 14+ 文件重编译
- **构建系统技术债**：拼写错误、冗余链接、失效的编译选项

### 1.2 目标

| 目标 | 度量标准 |
|------|----------|
| 算法层可独立使用 | 纯 C++ 项目链接 ct_core + ct_algorithm，无需 Qt/VTK |
| 编译隔离 | 修改 cloudview.h 不触发算法层重编译 |
| 目录结构专业 | 与 CloudCompare、PCL 等开源项目风格一致 |
| 可扩展 | 新增算法只需依赖 ct_core，新增 UI 工具无需关心 VTK |

## 2. 深度耦合分析

### 2.1 依赖链全景图

```
                     ┌─────────────────────────────┐
                     │        src/ (应用层)           │
                     │  mainwindow, tools, plugins  │
                     └──┬────┬────┬────┬────┬──────┘
                        │    │    │    │    │
                     ct_ui  ct_viz ct_io ct_py ct_algo
                        │    │    │    │    │
                        └┬───┘    │    │    │
                         │        │    │    │
                      ┌──┴────────┴────┴────┘
                      │
                   ct_core
                      │
               PCL + Eigen + Qt ← !!! 问题根源 !!!
```

**目标应为**：

```
                   ct_core → PCL + Eigen（无 Qt）
                   ct_algo → ct_core（无 Qt、无 VTK）
                   ct_viz  → ct_core + VTK + Qt
                   ct_ui   → ct_core + Qt
```

### 2.2 五个阻塞点详解

#### 阻塞点 1：Cloud 用 Qt 类型做成员变量

```cpp
// core/cloud.h — 当前
class Cloud {
    QString     m_id;                                     // Qt
    QString     m_type;                                   // Qt
    QFileInfo   m_info;                                   // Qt
    QString     m_current_color_mode;                     // Qt
    QMap<QString, std::vector<float>> m_scalar_cache;     // Qt
};
```

**影响**：Cloud 是整个项目最基础的数据结构，它的 Qt 依赖向下传递到所有模块。算法层调用 `cloud->id()` 返回 `QString`，被迫使用 Qt 类型。

**解决方案**：

```cpp
// core/cloud.h — 重构后
class Cloud {
    std::string m_id;                                     // std
    std::string m_type;                                   // std
    std::string m_filepath;                               // std（替代 QFileInfo）
    std::string m_current_color_mode;                     // std
    std::map<std::string, std::vector<float>> m_scalar_cache; // std
};
```

| 原类型 | 替换为 | 影响范围 |
|--------|--------|----------|
| `QString m_id` | `std::string m_id` | 所有调用 `cloud->id()` 的代码 |
| `QString m_type` | `std::string m_type` | 较少使用 |
| `QFileInfo m_info` | `std::string m_filepath` + 工具函数 | FileIO、CloudTree |
| `QString m_current_color_mode` | `std::string m_current_color_mode` | 较少使用 |
| `QMap<K,V> m_scalar_cache` | `std::map<K,V> m_scalar_cache` | Python bindings、标量操作 |

#### 阻塞点 2：CloudBlock 用 QMap

```cpp
// core/octree.h — 当前
class CloudBlock {
    QMap<QString, std::vector<float>> m_scalar_fields;    // Qt
};
```

**解决方案**：

```cpp
// core/octree.h — 重构后
class CloudBlock {
    std::map<std::string, std::vector<float>> m_scalar_fields; // std
};
```

#### 阻塞点 3：算法类继承 QObject

当前全部 7 个算法类都继承 QObject，使用 Qt 信号槽做结果回调：

```cpp
// modules/filters.h — 当前
class CT_EXPORT Filters : public QObject {
    Q_OBJECT
public slots:
    void apply(FilterType type, Cloud::Ptr cloud, ...);
signals:
    void filterResult(Cloud::Ptr cloud, float time);
};
```

**分析**：算法本身不需要 QObject。继承 QObject 的目的是：在 QThread 中运行算法，通过信号槽把结果回传给 UI。这是**异步通信的需求**，不是**算法逻辑的需求**。

**解决方案**：用 `std::function` 回调替代 Qt 信号槽，在 UI 层做 Qt 适配。

```cpp
// libs/algorithm/filter/filters.h — 重构后（纯 C++，无 Qt）
class Filters {
public:
    using ProgressCallback = std::function<void(int percent)>;
    using ResultCallback   = std::function<void(Cloud::Ptr result, float time)>;

    static void apply(FilterType type, Cloud::Ptr cloud,
                      const Params& params,
                      ResultCallback on_done = nullptr,
                      ProgressCallback on_progress = nullptr);
};
```

```cpp
// src/tool/filters.h — UI 层做 Qt 适配
class FiltersTool : public CustomDock {
    Q_OBJECT
    QThread m_thread;
    // 在 worker 线程中调用纯 C++ 算法
    // 算法回调中 emit Qt 信号给 UI
};
```

**改动清单**：

| 文件 | 当前 | 重构后 |
|------|------|--------|
| `filters.h/cpp` | `class Filters : public QObject` | `class Filters`（纯 C++） |
| `features.h/cpp` | `class Features : public QObject` | `class Features`（纯 C++） |
| `registration.h/cpp` | `class Registration : public QObject` | `class Registration`（纯 C++） |
| `keypoints.h/cpp` | `class Keypoints : public QObject` | `class Keypoints`（纯 C++） |
| `csffilter.h/cpp` | `class CSFFilter : public QObject` | `class CSFFilter`（纯 C++） |
| `vegfilter.h/cpp` | `class VegetationFilter : public QObject` | `class VegetationFilter`（纯 C++） |
| `distancecalculator.h/cpp` | `class DistanceCalculator : public QObject` | `class DistanceCalculator`（纯 C++） |

#### 阻塞点 4：exports.h 用 Qt 宏

```cpp
// core/exports.h — 当前
#include <QtCore/qglobal.h>
#define CT_EXPORT Q_DECL_EXPORT
#define CT_IMPORT Q_DECL_IMPORT
```

**解决方案**：用平台原生宏替代 Qt 宏：

```cpp
// core/exports.h — 重构后（无 Qt 依赖）
#if defined(_WIN32)
    #ifdef CT_BUILDING_CORE
        #define CT_EXPORT __declspec(dllexport)
    #else
        #define CT_EXPORT __declspec(dllimport)
    #endif
#else
    #define CT_EXPORT __attribute__((visibility("default")))
#endif
```

#### 阻塞点 5：UI 基类 include cloudview.h

```cpp
// widgets/customdialog.h — 当前
#include "core/cloudview.h"  // 拉入 VTK + PCL + Qt 整个栈
#include "cloudtree.h"

// widgets/customdock.h — 当前
#include "core/cloudview.h"  // 同上
#include "cloudtree.h"

// widgets/customtree.h — 当前
#include "core/cloudview.h"  // 一个树控件基类，为什么要依赖 VTK？
```

**结果**：全部 14 个工具/插件被迫传递依赖 VTK。

**解决方案**：用前向声明替代 include：

```cpp
// customdialog.h — 重构后
namespace ct { class CloudView; }   // 前向声明
namespace ct { class CloudTree; }   // 前向声明
namespace ct { class Console; }     // 前向声明

class CustomDialog : public QDialog {
protected:
    CloudView*  m_cloudview;   // 指针只需前向声明
    CloudTree*  m_cloudtree;
    Console*    m_console;
};
```

将 `#include "core/cloudview.h"` 移到 `.cpp` 文件中。这样修改 `cloudview.h` 不再触发 14 个工具/插件重编译。

### 2.3 其他耦合问题

#### `common.h` 是垃圾抽屉

```cpp
// core/common.h — 同时包含三类不相关的东西：
// 1. 数据结构：FieldInfo, MappingResult, TxtImportParams, TxtExportParams, DistanceParams
// 2. 工具函数：HSVtoRGB, getEulerAngles, getTransformation
// 3. PCL 头文件：5 个 pcl/common/*.h（但 .h 里并不调用 PCL 函数）
```

轻量级 UI 对话框（`txtimportdialog.h` 等）只为 `FieldInfo` 结构体 include 了 `common.h`，结果被迫编译了 5 个 PCL 头文件。

**解决方案**：拆分为三个文件：

```
libs/core/field_types.h    — FieldInfo, MappingResult, TxtImportParams 等（纯数据，无 PCL）
libs/core/algorithm_utils.h — HSVtoRGB, getEulerAngles 等（需要 Eigen/PCL，只在 .cpp 中 include）
libs/core/cloud_utils.h    — 其他 Cloud 工具函数
```

#### `csffilter.h` 泄漏第三方实现

```cpp
// modules/csffilter.h — 当前
#include "3rdparty/CSF/src/CSF.h"           // 第三方头文件暴露
#include "3rdparty/CSF/src/point_cloud.h"    // 第三方头文件暴露
```

**解决方案**：移到 `.cpp`，头文件只声明接口：

```cpp
// csffilter.h — 重构后
class CSFFilter {
public:
    struct Params { /* ... */ };
    static Cloud::Ptr apply(const Cloud::Ptr& cloud, const Params& params);
};

// csffilter.cpp
#include "3rdparty/CSF/src/CSF.h"  // 实现细节隐藏在 .cpp 中
```

#### `cloudtool/tool/registration.h` 直接依赖同层 `descriptor.h`

```cpp
// cloudtool/tool/registration.h
#include "cloudtool/tool/descriptor.h"  // 两个工具之间不应有直接依赖
```

**解决方案**：前向声明 + setter：

```cpp
class Descriptor;  // 前向声明
class Registration : public CustomDock {
    void setDescriptor(Descriptor* desc);
};
```

#### `modules/filters.h` 包含 14 个 PCL 模板实现头文件

`filters.h` 直接 include 了 `pcl/sample_consensus/impl/sac_model_*.hpp`（14 个模板实现），导致所有 include `filters.h` 的文件编译时间极长。

**解决方案**：将 PCL 模板 include 移到 `filters.cpp`，头文件只做前向声明。

### 3.3 独立使用算法的示例

```cpp
// 纯 C++ 命令行项目，无需 Qt/VTK
#include <cloud.h>
#include <filters.h>

int main() {
    auto cloud = std::make_shared<ct::Cloud>();
    // ... 填充点云 ...

    ct::Filters::ProgressCallback on_progress = [](int pct) {
        printf("Progress: %d%%\n", pct);

    auto result = ct::Filters::apply(
        ct::FilterType::VoxelGrid,
        cloud,
        {.leaf_size = 0.1f},
        nullptr,         // on_done（同步调用不需要）
        on_progress

    printf("Downsampled: %zu -> %zu points\n",
           cloud->size(), result->size());
    return 0;
}
```

### 4.3 算法类去 QObject 化

以 `Filters` 为例：

```cpp
// ===== Before (modules/filters.h) =====
class CT_EXPORT Filters : public QObject {
    Q_OBJECT
public slots:
    void apply(FilterType type, Cloud::Ptr cloud, float param1, ...);
signals:
    void filterResult(Cloud::Ptr cloud, float time);
    void progress(int percent);
};

// ===== After (libs/algorithm/filter/filters.h) =====
class Filters {
public:
    using ProgressCallback = std::function<void(int percent)>;

    struct Params {
        float param1 = 0;
        float param2 = 0;
        // ...
    };

    struct Result {
        Cloud::Ptr cloud;
        float time_ms = 0;
    };

    // 同步调用（纯 C++）
    static Result apply(FilterType type, const Cloud::Ptr& cloud,
                        const Params& params,
                        ProgressCallback on_progress = nullptr);

    // 异步调用（可选，内部用 std::thread）
    static std::future<Result> applyAsync(FilterType type,
                                          const Cloud::Ptr& cloud,
                                          const Params& params,
                                          ProgressCallback on_progress = nullptr);
};
```

**UI 层适配（src/tool/filters.h）**：

```cpp
// 工具类继承 QObject，负责 UI 和线程管理
class FiltersTool : public CustomDock {
    Q_OBJECT
    QThread m_thread;

    void onApply() {
        auto params = collectParams();
        auto cloud = m_cloudtree->currentCloud();

        // 在工作线程中调用纯 C++ 算法
        QFuture<ct::Filters::Result> future =
            QtConcurrent::run([=]() {
                return ct::Filters::apply(type, cloud, params);
            });

        // 用 QFutureWatcher 桥接到 Qt 信号槽
        auto* watcher = new QFutureWatcher<ct::Filters::Result>();
        connect(watcher, &QFutureWatcher<ct::Filters::Result>::finished, [=]() {
            auto result = watcher->result();
            emit filterDone(result.cloud, result.time_ms);
            watcher->deleteLater();
        });
        watcher->setFuture(future);
    }
};
```

## 6. 实施计划

### 第一阶段：代码去 Qt 化（核心改动）

**目标**：让 ct_core 和 ct_algorithm 可脱离 Qt 编译。

| 步骤 | 操作 | 改动量 |
|------|------|--------|
| 1 | `exports.h` 替换为平台原生宏 | 1 文件 |
| 2 | `cloud.h` 成员变量 `QString→std::string`，`QMap→std::map`，`QFileInfo→std::string` | 1 文件 + 所有调用处 |
| 3 | `octree.h` 中 `QMap→std::map` | 1 文件 + 调用处 |
| 4 | 7 个算法类去除 QObject 继承，改用回调 | 7 对 .h/.cpp |
| 5 | `common.h` 拆分为 `field_types.h` + `cloud_utils.h` | 3 文件 |
| 6 | UI 基类 `cloudview.h` include 改为前向声明 | 3 对 .h/.cpp |

**验证**：新建一个纯 C++ 项目，只 `find_package(PCL)` + `target_link_libraries(ct_core ct_algorithm)`，编译运行一个滤波操作。

### 第二阶段：目录重组

**目标**：按目标结构调整目录和文件归属。

| 步骤 | 操作 |
|------|------|
| 1 | `cloudtool/` → `src/` |
| 2 | `core/` → `libs/core/`，拆出 `cloudview.*` → `libs/viz/`，拆出 `fileio.*` → `libs/io/` |
| 3 | `modules/` → `libs/algorithm/`，按功能分子目录 |
| 4 | `widgets/` → `libs/ui/`，`common_ui/` 提升为平级 `dialog/` |
| 5 | `python/` → `libs/python/` |
| 6 | Python UI 组件从 `src/tool/` 移到 `src/python/` |
| 7 | 删除 `ui_*.h` 自动生成文件 |
| 8 | 更新所有 CMakeLists.txt |

**验证**：全量编译通过，功能回归测试。

### 第三阶段：CMake 清理 + 细节优化

| 步骤 | 操作 |
|------|------|
| 1 | 全局 `include_directories()` 迁移为 target-based |
| 2 | 修复所有拼写错误和冗余链接 |
| 3 | `csffilter.h` 隐藏 CSF 头文件 |
| 4 | `registration.h` 前向声明 `Descriptor` |
| 5 | `filters.h` 的 PCL 模板 include 移到 `.cpp` |
| 6 | 清理 `camera/` 死代码 |

**验证**：编译时间对比（预期减少 30%+），算法库可独立编译。

## 7. 风险评估

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| `QString→std::string` 导致中文路径/文件名问题 | 中 | 文件加载失败 | 使用 `std::filesystem::path`（C++17），内部用 UTF-8 |
| 算法去 QObject 后 UI 层异步调度需要重写 | 高 | 每个 Tool/Plugin 的线程模型需要调整 | 用 `QtConcurrent::run` + `QFutureWatcher` 适配 |
| `Cloud::id()` 类型变更影响 Python bindings | 中 | Python API 不兼容 | 在 bindings 层做 `std::string↔str` 适配 |
| Cloud/FileIO friend 拆分导致性能回退 | 低 | 大文件加载变慢 | 新接口保持零拷贝语义，benchmark 对比 |
| 目录移动导致 git 历史断裂 | 中 | 代码追溯困难 | 使用 `git mv`，或 `git log --follow` |
